PROJECT MANAGEMENT

Project management is the application of knowledge, skills, tools, and techniques to project activities to meet project requirements. The PMBOK (Project Management Body of Knowledge) framework defines structured approaches to planning, executing, monitoring, and closing projects.

FIVE PROCESS GROUPS

1

INITIATING

Define project scope, objectives, and stakeholders. Develop project charter and identify initial requirements. Obtain authorization to start the project.

Key Deliverables:
  • • Project Charter with business case and success criteria
  • • Stakeholder Register identifying all parties
  • • High-level requirements and constraints
2

PLANNING

Establish project scope, define activities, sequence them, estimate resources and duration, develop schedule, estimate costs, determine budget, plan quality, and identify risks.

Planning Activities:
• Scope Management Plan
• Schedule Management Plan
• Cost Management Plan
• Quality Management Plan
• Resource Management Plan
• Risk Management Plan
3

EXECUTING

Direct and manage project work, manage team, implement quality assurance, acquire resources, conduct procurements, and manage stakeholder engagement.

Execution Focus:
  • • Coordinate people and resources to carry out the plan
  • • Implement approved changes and corrective actions
  • • Conduct quality assurance and team development
  • • Distribute information and manage stakeholder expectations
4

MONITORING & CONTROLLING

Track, review, and regulate progress; identify variances from plan; implement corrective actions; control scope, schedule, costs, quality, and risks.

Control Processes:
• Monitor Project Work
• Perform Integrated Change Control
• Validate & Control Scope
• Control Schedule & Costs
• Control Quality
• Monitor Risks
5

CLOSING

Finalize all activities, obtain formal acceptance, conduct post-project reviews, document lessons learned, release resources, and archive project information.

Closure Activities:
  • • Obtain formal acceptance and sign-off from stakeholders
  • • Complete financial closure and final payments
  • • Document lessons learned for future projects
  • • Release project team and resources
  • • Archive all project documentation

KEY SUCCESS FACTORS

✓ Clear project objectives and success criteria
✓ Strong executive sponsorship and stakeholder engagement
✓ Skilled project team with defined roles
✓ Realistic scheduling and resource allocation
✓ Effective communication and risk management
✓ Regular monitoring and adaptive control

GOVERNANCE STRUCTURE & ROLES

Effective IT governance ensures that IT investments support business objectives, risks are managed appropriately, and resources are used responsibly. Clear roles and responsibilities are essential for accountability.

SC

Steering Committee

Provides strategic direction, approves major decisions, allocates resources, and resolves escalated issues. Typically includes senior executives and business leaders.

PS

Project Sponsor

Champions the project, provides funding, removes organizational barriers, and ensures alignment with business strategy. Primary liaison to executive leadership.

PM

Project Manager

Leads day-to-day project activities, manages team, coordinates tasks, monitors progress, controls budget and schedule, manages risks and issues.

BO

Business Owner

Defines business requirements, validates deliverables, makes business decisions, ensures benefits realization, and owns the solution post-implementation.

TL

Technical Lead

Provides technical direction, makes architectural decisions, ensures technical quality, resolves technical issues, and mentors development team.

QA

Quality Assurance

Defines quality standards, conducts testing, verifies requirements, ensures compliance with processes, and validates deliverables before release.

SYSTEM DEVELOPMENT LIFE CYCLE (SDLC)

SDLC is a structured process for planning, creating, testing, and deploying information systems. It provides a systematic approach that ensures quality, meets user needs, and is delivered on time and within budget.

1

PLANNING & FEASIBILITY STUDY

Define project scope, conduct feasibility analysis (technical, operational, economic, legal), identify stakeholders, establish high-level requirements, and obtain project approval.

Deliverables: Project charter, feasibility report, preliminary requirements
2

REQUIREMENTS ANALYSIS

Gather detailed functional and non-functional requirements through interviews, surveys, workshops. Document use cases, user stories, and acceptance criteria. Validate requirements with stakeholders.

Deliverables: SRS document, use cases, data flow diagrams, requirement traceability matrix
3

SYSTEM DESIGN

Create high-level design (architecture, technology stack, integration points) and detailed design (database schema, API specifications, UI/UX mockups, security architecture).

Deliverables: System design document, database design, architecture diagrams, UI mockups
4

DEVELOPMENT / IMPLEMENTATION

Code the system according to design specifications, conduct unit testing, perform code reviews, manage version control, and document code for maintainability.

Deliverables: Source code, unit test results, technical documentation, version control logs
5

TESTING

Conduct integration testing, system testing, user acceptance testing (UAT), performance testing, and security testing. Log defects and verify fixes.

Deliverables: Test plans, test cases, test results, defect reports, UAT sign-off
6

DEPLOYMENT

Prepare production environment, migrate data, deploy application, conduct user training, provide documentation, and plan rollback procedures if needed.

Deliverables: Deployment plan, user manuals, training materials, go-live checklist
7

MAINTENANCE & SUPPORT

Monitor system performance, fix bugs, implement enhancements, provide user support, apply security patches, and conduct periodic reviews for optimization.

Deliverables: Maintenance logs, change requests, performance reports, patch documentation

SOFTWARE DEVELOPMENT METHODS

Different software development methodologies offer varying approaches to project execution, each with distinct advantages and trade-offs. Selecting the right methodology depends on project requirements, team capabilities, and organizational context.

WATERFALL METHODOLOGY

Sequential, linear approach where each phase must be completed before the next begins. Best suited for projects with stable, well-understood requirements and regulatory constraints.

✓ ADVANTAGES

  • • Clear structure and comprehensive documentation
  • • Easy to understand and manage
  • • Well-defined milestones and deliverables
  • • Suitable for fixed-scope projects
  • • Good for regulatory/compliance requirements

✗ DISADVANTAGES

  • • Inflexible to requirement changes
  • • Late discovery of issues in testing phase
  • • No working software until late in project
  • • Risk of misalignment with user needs
  • • High cost of change late in cycle

AGILE METHODOLOGY

Iterative, incremental approach emphasizing flexibility, collaboration, and continuous delivery. Development occurs in short sprints (1-4 weeks) with regular feedback and adaptation.

FOUR CORE AGILE VALUES

Individuals and interactions
over processes and tools
Working software
over comprehensive documentation
Customer collaboration
over contract negotiation
Responding to change
over following a plan

✓ ADVANTAGES

  • • Flexible and adaptive to changes
  • • Continuous delivery of value to customers
  • • Early and frequent user feedback
  • • Better risk management through iterations
  • • Higher customer satisfaction
  • • Team collaboration and morale

✗ DISADVANTAGES

  • • Less predictable timelines and costs
  • • Requires experienced, self-organizing team
  • • Can lack comprehensive documentation
  • • Needs active stakeholder involvement
  • • Difficult with distributed teams

SCRUM FRAMEWORK

Most popular Agile framework with defined roles, events, and artifacts. Work is organized in time-boxed sprints with daily coordination and regular inspect-and-adapt cycles.

PO

PRODUCT OWNER

  • • Defines product features
  • • Prioritizes backlog
  • • Accepts/rejects work
  • • Ensures ROI
SM

SCRUM MASTER

  • • Facilitates Scrum process
  • • Removes impediments
  • • Coaches team
  • • Protects from distractions
DT

DEVELOPMENT TEAM

  • • Cross-functional (3-9 members)
  • • Self-organizing
  • • Designs, builds, tests
  • • Delivers increments

SCRUM EVENTS

Sprint Planning
Team selects work from product backlog and plans sprint execution (time-boxed 8h for 1-month sprint)
Daily Standup
15-minute daily sync: What did I do? What will I do? Any blockers?
Sprint Review
Demonstrate completed work to stakeholders, gather feedback (time-boxed 4h for 1-month sprint)
Sprint Retrospective
Team reflects on process, identifies improvements for next sprint (time-boxed 3h for 1-month sprint)

DEVOPS

Cultural and technical movement combining software development (Dev) and IT operations (Ops) to shorten development cycles and provide continuous delivery with high software quality.

KEY PRACTICES

Continuous Integration (CI)

Frequent code commits, automated builds and tests

Continuous Deployment (CD)

Automated release pipeline from commit to production

Infrastructure as Code (IaC)

Manage infrastructure through code and version control

Monitoring & Logging

Continuous monitoring of applications and infrastructure

BENEFITS

  • • Faster time to market
  • • Improved collaboration between teams
  • • Higher deployment frequency
  • • Lower failure rate of releases
  • • Faster mean time to recovery (MTTR)
  • • Better quality and reliability

HARDWARE/SOFTWARE ACQUISITION & MEITY MODEL RFP

The Ministry of Electronics and Information Technology (MEITY) provides Model Request for Proposal (RFP) documents to standardize procurement processes for government departments, ensuring transparency, fairness, and value for money.

ACQUISITION PROCESS

1

NEEDS ASSESSMENT

Identify business requirements, technical specifications, budget constraints, and success criteria. Conduct gap analysis of current vs. desired state. Document functional and non-functional requirements.

2

MARKET RESEARCH

Research available solutions, vendors, industry standards, and pricing models. Conduct proof-of-concept if necessary. Analyze total cost of ownership (TCO) including licensing, maintenance, and support.

3

RFP PREPARATION

Draft RFP document including scope, technical requirements, evaluation criteria, terms and conditions, and timelines. Follow MEITY Model RFP guidelines to ensure compliance and standardization.

4

VENDOR SELECTION

Issue RFP, conduct pre-bid meetings, evaluate proposals against criteria, shortlist vendors, conduct presentations/demos, perform due diligence and reference checks.

5

CONTRACT NEGOTIATION

Negotiate pricing, payment terms, service levels, warranties, support terms, and exit clauses. Finalize contract with legal review ensuring compliance with procurement policies.

6

IMPLEMENTATION & ACCEPTANCE

Deploy solution, conduct user acceptance testing, verify compliance with requirements, obtain sign-off, and release payment as per agreed milestones.

KEY ELEMENTS OF MEITY MODEL RFP

Technical Requirements
  • • Functional specifications and use cases
  • • Performance requirements (response time, throughput)
  • • Security and compliance needs
  • • Integration requirements with existing systems
  • • Scalability and availability requirements
Commercial Terms
  • • Pricing structure and payment terms
  • • Warranty and post-warranty support
  • • Penalties and liquidated damages
  • • Service level agreements (SLAs)
  • • Intellectual property rights

IT SERVICES MANAGEMENT (ITSM) & ITIL

ITSM is a set of policies, processes, and procedures for implementing, delivering, and managing IT services. ITIL (Information Technology Infrastructure Library) is the most widely adopted framework for ITSM, providing best practices for aligning IT services with business needs.

ITIL 4 SERVICE VALUE SYSTEM

Guiding Principles: Focus on value, start where you are, progress iteratively with feedback, collaborate and promote visibility, think and work holistically, keep it simple and practical, optimize and automate.

CORE ITSM PROCESSES

Incident Management
Restore normal service operation as quickly as possible with minimal impact on business. Log, categorize, prioritize, diagnose, and resolve incidents. Track KPIs like MTTR (Mean Time to Resolve).
Problem Management
Identify root causes of incidents and prevent recurrence. Manage known errors and implement permanent fixes through change management. Conduct proactive problem identification.
Change Management
Control the lifecycle of all changes to minimize disruption. Assess risks, obtain approvals, coordinate implementation, and conduct post-implementation review. Maintain Change Advisory Board (CAB).
Service Request Management
Handle user requests for information, access, or standard services. Provide self-service portal and automate fulfillment where possible. Define request models and workflows.
Configuration Management
Maintain information about configuration items (CIs) and their relationships in Configuration Management Database (CMDB). Ensure accuracy and completeness. Support impact analysis.
Release Management
Plan, schedule, and control the movement of releases to test and live environments. Ensure release integrity and traceability. Coordinate with change management.
Service Level Management
Define, document, agree, monitor, and review service levels. Ensure services meet agreed targets and continuously improve. Manage SLAs, OLAs (Operational Level Agreements), and UCs (Underpinning Contracts).
Availability Management
Ensure IT services meet agreed availability targets. Measure, monitor, and optimize availability, reliability, and maintainability. Conduct risk assessments and implement redundancy.
Capacity Management
Ensure current and future capacity meets business requirements in cost-effective manner. Monitor utilization, forecast needs, and plan capacity. Covers business, service, and component capacity management.
IT Service Continuity Management (ITSCM)
Manage risks that could seriously impact IT services. Ensure recovery of critical IT services within agreed timeframes. Align with Business Continuity Planning (BCP).

BUSINESS CONTINUITY PLANNING (BCP)

Business Continuity Planning ensures that critical business functions can continue during and after a disaster. It encompasses disaster recovery, crisis management, and business resumption planning to minimize operational disruption.

KEY CONCEPTS

RTO

Recovery Time Objective

Maximum acceptable time to restore a business process after disruption. Defines how quickly systems must be recovered. Example: Email must be restored within 4 hours.

RPO

Recovery Point Objective

Maximum acceptable amount of data loss measured in time. Determines backup frequency and data replication strategy. Example: Database can lose max 1 hour of transactions.

MTD

Maximum Tolerable Downtime

Maximum time a business process can be down before organization suffers unacceptable consequences. Sets upper bound for RTO.

BCP DEVELOPMENT PROCESS

1

BUSINESS IMPACT ANALYSIS (BIA)

Identify critical business functions, assess impact of disruptions, determine RTO and RPO for each function, prioritize recovery efforts. Quantify financial and operational impacts.

Outputs: Critical function list, impact assessments, RTO/RPO requirements, recovery priorities
2

RISK ASSESSMENT

Identify potential threats (natural disasters, cyber attacks, equipment failure), assess likelihood and impact, develop mitigation strategies. Consider internal and external risks.

Common Threats: Fire, flood, earthquake, power outage, cyber attack, pandemic, supply chain disruption
3

STRATEGY DEVELOPMENT

Define recovery strategies for people, processes, technology, and facilities. Select backup sites, data replication methods, and communication plans. Consider cost vs. recovery speed.

Recovery Strategies: Hot site (< 24h), Warm site (1-3 days), Cold site (> 1 week), Cloud-based DR
4

PLAN DEVELOPMENT

Document detailed procedures for response, recovery, and resumption. Define roles, responsibilities, contact lists, and escalation procedures. Create runbooks for critical scenarios.

Plan Components: Emergency response, damage assessment, recovery procedures, communication protocols
5

TESTING & TRAINING

Conduct tabletop exercises, simulation tests, and full-scale drills. Train staff on procedures, test recovery capabilities, identify gaps. Document lessons learned.

6

MAINTENANCE & REVIEW

Regularly review and update plans, conduct annual tests, incorporate lessons learned, update for organizational changes. Maintain plan currency and relevance.

COMMON BCP TESTING METHODS

1.
Checklist Review: Distribute plans for review and feedback. Low cost, minimal disruption.
2.
Tabletop Exercise: Walk through scenarios in conference room setting. Discuss roles and responses.
3.
Simulation Test: Execute portions of plan without actual failover. Test specific components.
4.
Parallel Test: Activate backup site while primary remains operational. Verify recovery capabilities.
5.
Full Interruption Test: Complete failover to backup site (highest risk, highest confidence). Rarely performed.

ISO 27000 SERIES (INFORMATION SECURITY)

ISO/IEC 27000 family provides internationally recognized standards for establishing, implementing, maintaining, and continually improving an Information Security Management System (ISMS).

ISO/IEC 27001:2022

Information Security Management Systems - Requirements. The main standard specifying requirements for establishing, implementing, maintaining, and continually improving an ISMS.

Annex A: 93 Controls in 4 Categories
• Organizational (37 controls): Policies, roles, asset management
• People (8 controls): Screening, training, awareness
• Physical (14 controls): Perimeter security, physical access
• Technological (34 controls): Access control, cryptography, network security
Key Change (2022): Reduced from 114 to 93 controls, reorganized from 14 to 4 categories for better alignment with modern security practices. Transition deadline: October 31, 2025.

ISO/IEC 27002:2022

Information Security Controls - Code of Practice. Provides guidance on selecting, implementing, and managing information security controls based on the organization's information security risk environment.

Detailed implementation guidance for the 93 controls in ISO 27001 Annex A. Essential reference for ISMS practitioners.

ISO/IEC 27005

Information Security Risk Management. Provides guidelines for information security risk assessment and treatment, supporting ISO 27001 requirements.

Risk Assessment: Context establishment, risk identification, risk analysis, risk evaluation
Risk Treatment: Risk modification, risk retention, risk avoidance, risk sharing

OTHER KEY STANDARDS

ISO/IEC 27018

ISO/IEC 38500 (IT GOVERNANCE)

ISO/IEC 38500 provides guiding principles for members of governing bodies of organizations on the effective, efficient, and acceptable use of IT within their organizations.

SIX PRINCIPLES OF IT GOVERNANCE

1
RESPONSIBILITY

Individuals and groups have clearly defined responsibilities for IT. Those making decisions are accountable for them.

2
STRATEGY

IT strategy supports and enables the organization's strategy. IT plans meet current and ongoing needs.

3
ACQUISITION

IT acquisitions are made for valid reasons, with appropriate analysis, clear decision-making, and balanced benefits, risks, and costs.

4
PERFORMANCE

IT is fit for purpose, supports the organization, provides services, quality, and adequate service levels to meet requirements.

5
CONFORMANCE

IT complies with mandatory legislation and regulations. Policies and practices are defined, implemented, and enforced.

6
HUMAN BEHAVIOR

IT policies, practices, and decisions respect human behavior including current and future needs of all people involved.

INFORMATION TECHNOLOGY ACT, 2000

The IT Act 2000 provides legal recognition to electronic transactions, digital signatures, and cybercrime penalties in India. Amended in 2008 to address emerging cyber threats and data protection concerns.

LEGAL RECOGNITION

Electronic Records (Section 4): Legal recognition to electronic records and documents. Where law requires information in writing, electronic form is acceptable.
Digital Signatures (Section 3): Authentication of electronic records through digital signatures. Asymmetric crypto system with private/public key pairs.
Electronic Governance (Section 6-7): Filing, creation, retention of documents in electronic form by government. Official publications in electronic gazette.

CERTIFYING AUTHORITIES & PKI

Controller of Certifying Authorities (CCA) licenses and regulates Certifying Authorities (CAs) to issue Digital Signature Certificates.

Duties of CAs:
  • • Verify identity of applicants
  • • Issue Digital Signature Certificates
  • • Maintain certificate repository
  • • Publish and maintain Certificate Revocation List (CRL)
Duties of Subscribers:
  • • Generate key pair securely
  • • Protect private key
  • • Notify compromise immediately
  • • Provide accurate information

CYBERCRIME PROVISIONS (KEY SECTIONS)

Section 43: Penalty for unauthorized access, data alteration, virus introduction. Compensation up to ₹5 crore.
Section 66: Hacking with computer system - imprisonment up to 3 years and/or fine up to ₹5 lakh.
Section 66B: Dishonest receipt of stolen computer/communication device - 3 years imprisonment and/or ₹1 lakh fine.
Section 66C: Identity theft using password/digital signature - 3 years imprisonment and/or ₹1 lakh fine.
Section 66D: Cheating by personation using computer resource - 3 years imprisonment and/or ₹1 lakh fine.
Section 66F: Cyber terrorism affecting unity, integrity, security of India - life imprisonment.
Section 67: Publishing obscene information in electronic form - 3 years (first conviction), 5 years (subsequent) + fine.
Section 72: Breach of confidentiality and privacy - 2 years imprisonment and/or ₹1 lakh fine.

INTERMEDIARY LIABILITY (SECTION 79)

Intermediaries (ISPs, hosting providers, social media) not liable if they exercise due diligence and observe guidelines.

Conditions for Safe Harbor:
  • • Function is limited to providing access/transmission
  • • Does not initiate transmission, select receiver, or modify information
  • • Observes due diligence and prescribed guidelines
  • • Removes unlawful content upon actual knowledge or notification

DATA PROTECTION (SECTION 43A)

Body corporate maintaining sensitive personal data must implement reasonable security practices and procedures. Failure causing wrongful loss makes them liable for compensation.

Sensitive Personal Data:
  • • Passwords
  • • Financial information
  • • Health conditions
  • • Sexual orientation
  • • Medical records
  • • Biometric information
Reasonable Security Practices:
  • • IS 27001 certification
  • • Documented security policy
  • • Periodic security audits
  • • Data encryption
  • • Access controls

AADHAAR ACT, 2016

The Aadhaar (Targeted Delivery of Financial and Other Subsidies, Benefits and Services) Act, 2016 provides legal backing to the Aadhaar unique identification system, enabling use for authentication and delivery of subsidies/services.

AADHAAR ECOSYSTEM

Unique Identification Authority of India (UIDAI):

Statutory authority responsible for Aadhaar enrollment, authentication, and update infrastructure.

Aadhaar Number:

12-digit unique identity number issued to residents based on biometric and demographic data.

Biometric Information:

Photograph, fingerprints, and iris scans collected during enrollment for authentication.

Authentication:

Process of verifying identity using Aadhaar number and biometric/OTP.

KEY PROVISIONS

Section 3: Every resident has right to obtain Aadhaar number. Enrollment is voluntary (except for subsidies).
Section 7: Government can require Aadhaar for subsidies, benefits, and services from Consolidated Fund of India.
Section 29: Use of Aadhaar - Core biometric info can only be used for authentication. Cannot be shared except by court order.
Section 32: Security and confidentiality - UIDAI shall ensure security, safeguard, and confidentiality of identity info and authentication records.
Section 33: Access to biometric info restricted. Only with consent or court order. Authentication records retained max 6 months.

PRIVACY & SECURITY PROVISIONS

  • • Core biometric information cannot be shared or published
  • • Authentication records retained for max 6 months (Regulation 26)
  • • Encrypted transmission and storage of identity information
  • • Penalties for unauthorized access (₹10 lakh fine + 3 years imprisonment)
  • • Penalty for disclosure of identity information (₹1 lakh fine + 3 years imprisonment)
  • • Virtual ID and limited KYC for privacy-preserving authentication

KEY REGULATIONS

Enrollment Regulations:

Process for enrollment, update, de-duplication. Registrar and enrollment agency roles.

Authentication Regulations:

AUA (Authentication User Agency) and ASA (Authentication Service Agency) licensing and obligations.

Data Security Regulations:

Technical and organizational measures, encryption standards, access controls, audit requirements.

Sharing of Information Regulations:

Conditions under which information can be disclosed. Consent requirements.

6-Step Risk-Based Audit Planning Process

1

Identify Audit Universe

Catalog all IT systems, applications, databases, networks, and IT processes within audit scope

2

Assess Inherent Risk

Evaluate risk based on criticality, complexity, technology changes, previous audit findings, regulatory requirements

3

Evaluate Control Environment

Review existing IT controls, governance structures, policies, procedures, and control effectiveness

4

Calculate Residual Risk

Determine risk remaining after considering controls: Residual Risk = Inherent Risk - Control Effectiveness

5

Prioritize Audit Areas

Rank systems/processes by residual risk level - High, Medium, Low - to create audit schedule

6

Develop Audit Plan

Create multi-year audit plan allocating resources to highest-risk areas, define scope, objectives, timing

  • ▪ Regulatory & compliance requirements
  • ▪ System complexity & technology stack
  • ▪ Integration with other systems
  • ▪ Change frequency & stability
  • ▪ Historical audit findings & issues
  • ▪ Threat landscape & cyber risks

IT CONTROLS FRAMEWORK

General IT Controls vs Application Controls

General IT Controls (ITGC)

Controls that apply across all IT systems and provide the foundation for application controls to function effectively

1. Access Controls

  • • User authentication & authorization
  • • Role-based access control (RBAC)
  • • Privileged access management
  • • Access reviews & recertification

2. Change Management

  • • Change request & approval process
  • • Development & testing procedures
  • • Release management & deployment
  • • Emergency change procedures

3. Computer Operations

  • • Job scheduling & monitoring
  • • Batch processing controls
  • • System performance monitoring
  • • Incident & problem management

4. Backup & Recovery

  • • Backup schedules & procedures
  • • Off-site storage & protection
  • • Recovery testing & validation
  • • Business continuity planning

5. Physical & Environmental

  • • Data center security
  • • Environmental controls (HVAC)
  • • Power & UPS management
  • • Fire suppression systems

Application Controls

Controls embedded within business application systems to ensure completeness, accuracy, and validity of transactions

1. Input Controls

  • • Data validation & edit checks
  • • Format & range checks
  • • Duplicate detection
  • • Batch totals & reconciliation
  • • Error handling & correction

2. Processing Controls

  • • Run-to-run totals
  • • Sequence checking
  • • Transaction logs & audit trails
  • • Calculation accuracy checks
  • • Database integrity controls

3. Output Controls

  • • Output reconciliation
  • • Report distribution controls
  • • Output completeness checks
  • • Print/file security
  • • User review & approval

4. Interface Controls

  • • Data transmission validation
  • • Reconciliation between systems
  • • Error handling & retry logic
  • • Interface monitoring

5. Master Data Controls

  • • Master data creation approval
  • • Change authorization workflow
  • • Segregation of duties
  • • Periodic review & cleanup

Control Objectives (CAAAV Framework)

C

Completeness

All transactions are captured & processed

A

Accuracy

Transactions processed correctly without errors

A

Authorization

Only approved transactions are processed

A

Auditability

Audit trails & logs are maintained

V

Validity

Transactions are legitimate & authorized

Relationship Between ITGC & Application Controls

ITGCs provide the foundation upon which application controls rely. If ITGCs are weak or ineffective, application controls cannot be fully relied upon. For example:

  • → Weak access controls (ITGC) can allow unauthorized changes to application validation rules (Application Control)
  • → Poor change management (ITGC) can result in untested code modifications that break input controls (Application Control)
  • → Inadequate backup procedures (ITGC) can render application audit trails (Application Control) unrecoverable in disaster scenarios

DATA ANALYTICS & CAATs

Computer-Assisted Audit Tools & Techniques

What are CAATs?

Computer-Assisted Audit Tools and Techniques (CAATs) are automated tools that enable auditors to analyze large volumes of data efficiently, identify anomalies, test controls, and perform substantive testing procedures that would be impractical manually.

ACL Analytics

Specialized audit analytics software for data extraction, analysis, and reporting

Key Features:

  • • Import data from any source
  • • 100% data coverage analysis
  • • Statistical sampling
  • • Duplicate detection
  • • Gap analysis & sequencing
  • • Aging & stratification
  • • Benford's Law analysis
  • • Custom scripting (ACL Script)
IDEA (CaseWare)

Interactive Data Extraction & Analysis - powerful audit analytics platform

Key Features:

  • • Direct database connectivity
  • • Join & relate datasets
  • • Visual analytics & charts
  • • Automated report generation
  • • Time series analysis
  • • Fuzzy matching algorithms
  • • Risk scoring models
  • • Continuous monitoring
SQL Queries

Structured Query Language for direct database interrogation

Common Audit Uses:

  • • SELECT for data extraction
  • • WHERE for filtering records
  • • JOIN for combining tables
  • • GROUP BY for aggregation
  • • COUNT, SUM for calculations
  • • HAVING for aggregate filters
  • • Subqueries for complex tests
  • • UNION for combining results

Common Data Analytics Tests in Auditing

Compliance Testing:

  • → Duplicate Detection: Identify duplicate invoices, payments, employee records
  • → Gap Analysis: Find missing check numbers, invoice sequences, transaction IDs
  • → Authorization Violations: Transactions exceeding approval limits
  • → Segregation of Duties: Users performing incompatible functions
  • → Policy Compliance: Transactions violating business rules

Substantive Testing:

  • → Aging Analysis: Stratify receivables/payables by age buckets
  • → Stratification: Group data by value ranges for focused testing
  • → Statistical Sampling: Select representative samples for detailed review
  • → Benford's Law: Detect anomalies in number distributions (fraud indicator)
  • → Trend Analysis: Identify unusual patterns, spikes, or variances

Continuous Auditing

Automated, real-time or near-real-time audit procedures that provide continuous assurance over transactions and controls

Key Characteristics:

  • • Automated data extraction & analysis
  • • Real-time exception reporting
  • • Continuous control monitoring
  • • Embedded audit modules in applications
  • • Dashboard-based alerting
  • • Shift from periodic to continuous assurance

Benefits:

  • ✓ Timely detection of issues
  • ✓ Improved audit coverage
  • ✓ Reduced audit cycle time
  • ✓ Proactive risk management
  • ✓ Cost efficiency

Continuous Monitoring

Management-driven process for regular monitoring of controls and transactions to ensure operational effectiveness

Implementation Approach:

  • ① Identify key risks & controls
  • ② Define monitoring KPIs & thresholds
  • ③ Automate data collection & analysis
  • ④ Configure exception alerts
  • ⑤ Establish investigation workflows
  • ⑥ Monitor & refine indicators

Example Monitors:

ENCRYPTION & CRYPTOGRAPHY

Protecting Data Confidentiality & Integrity

Symmetric Encryption

Same key for encryption/decryption. Fast, efficient for bulk data.

Algorithms:

  • • AES - 128/192/256-bit, industry standard
  • • DES - 56-bit, deprecated
  • • 3DES - 168-bit, legacy systems

Use: Data at rest, VPNs, disk encryption

Asymmetric Encryption

Public-private key pair. Slower, solves key distribution.

Algorithms:

  • • RSA - 2048/4096-bit
  • • ECC - Smaller keys, same security
  • • Diffie-Hellman - Key exchange

Use: Digital signatures, SSL/TLS, email encryption

Hashing Algorithms

MD5

128-bit, broken

DEPRECATED

SHA-256

256-bit, standard

RECOMMENDED

SHA-512

512-bit, high security

FUTURE-PROOF

Public Key Infrastructure (PKI)

Framework for managing digital certificates and public-key encryption.

PKI Components:

1.
Certificate Authority (CA) - Issues and revokes certificates
2.
Registration Authority (RA) - Verifies user identity
3.
Digital Certificates (X.509) - Bind public keys to identities
4.
Certificate Repository - Stores certificates
5.
CRL/OCSP - Certificate revocation checking

Use Cases:

  • ✓ SSL/TLS (HTTPS) - Website security
  • ✓ Email encryption (S/MIME, PGP)
  • ✓ Code signing - Software authenticity
  • ✓ VPN authentication - Secure tunnels
  • ✓ Document signing - Legal documents
  • ✓ Smart card authentication

Encryption Best Practices

  • ✓ Use industry-standard algorithms (AES-256, RSA-2048+)
  • ✓ Never create custom cryptography
  • ✓ Implement proper key management lifecycle
  • ✓ Use strong random number generators
  • ✓ Encrypt data at rest AND in transit
  • ✓ Regularly rotate encryption keys
  • ✓ Protect keys with HSM (Hardware Security Module)
  • ✓ Implement perfect forward secrecy (PFS)
  • ✓ Use authenticated encryption (GCM mode)

Common Encryption Mistakes

  • ✗ Using deprecated algorithms (DES, MD5, SHA-1)
  • ✗ Hardcoding keys in source code
  • ✗ Insufficient key length (RSA < 2048 bits)
  • ✗ Not validating certificates properly
  • ✗ Storing passwords in reversible encryption
  • ✗ Ignoring certificate expiration
  • ✗ Using ECB mode (use CBC/GCM instead)
  • ✗ No salt for password hashing
  • ✗ Reusing IVs (Initialization Vectors)

Key Management Lifecycle

1. Generation

Create using secure RNG

2. Distribution

Secure key exchange

3. Storage

HSM, key vault

4. Rotation

Periodic updates

5. Destruction

Secure deletion

OWASP TOP 10 (2021)

Critical Web Application Security Risks

A01

Broken Access Control

Users act outside intended permissions

Example:

Changing URL from /user/123 to /user/124 to view others' data

Mitigation:

Implement server-side authorization checks, deny by default

A02

Cryptographic Failures

Sensitive data exposure via weak crypto

Example:

Transmitting passwords over HTTP, storing passwords in plain text

Mitigation:

Use TLS, encrypt sensitive data, strong algorithms (AES-256)

A03

Injection

SQL, command, LDAP injection attacks

Example:

SQL: ' OR '1'='1 -- bypasses authentication

Mitigation:

Use parameterized queries, input validation, ORM frameworks

A04

Insecure Design

Missing or ineffective control design

Example:

No rate limiting on password reset, allowing brute force

Mitigation:

Threat modeling, secure design principles, defense in depth

A05

Security Misconfiguration

Insecure defaults, missing patches

Example:

Default admin/admin credentials, directory listing enabled

Mitigation:

Hardening guides, automated configuration management, patch regularly

A06

Vulnerable Components

Libraries with known vulnerabilities

Example:

Using Log4j with Log4Shell vulnerability (CVE-2021-44228)

Mitigation:

Dependency scanning, SBOM, keep components updated

A07

ID & Auth Failures

Broken authentication mechanisms

Example:

Session fixation, weak password policy, credential stuffing

Mitigation:

Implement MFA, secure session management, strong passwords

A08

Software & Data Integrity

Unverified updates, supply chain attacks

Example:

SolarWinds supply chain attack, insecure CI/CD pipelines

Mitigation:

Code signing, verify digital signatures, secure CI/CD

A09

Logging & Monitoring Failures

Insufficient logging and alerting

Example:

No logging of failed login attempts, no alerting on anomalies

Mitigation:

Implement SIEM, log security events, real-time alerts

A10

Server-Side Request Forgery

SSRF - accessing internal resources

Example:

Fetch http://169.254.169.254/latest/meta-data to steal AWS credentials

Mitigation:

Validate and sanitize URLs, network segmentation, deny lists

Social Engineering Attack Types

Phishing

Fraudulent emails requesting credentials or sensitive info. Often impersonates trusted entities.

Spear Phishing

Targeted phishing using personal information about specific individuals or organizations.

Vishing (Voice Phishing)

Phone calls impersonating IT support, bank representatives, or government agencies.

Smishing (SMS Phishing)

Text messages with malicious links or requests for sensitive information.

Pretexting

Fabricated scenario to gain trust (e.g., posing as contractor, auditor, or new employee).

Baiting

Offering something enticing (free software, USB drive) that contains malware.

Quid Pro Quo

Offering a service in exchange for information (e.g., "IT support" helping with fake issue).

Tailgating/Piggybacking

Following authorized person into secure area without proper authentication.

Organizing & Protecting Information Assets

Data Classification Framework

Systematic categorization of data based on sensitivity, value, and criticality to determine appropriate protection levels.

1

Public

Information intended for public disclosure. No harm if released.

Examples:

  • • Marketing materials
  • • Press releases
  • • Public websites
  • • Published reports

Protection: Standard web security

2

Internal

For internal use only. Limited distribution within organization.

Examples:

  • • Internal policies
  • • Department budgets
  • • Employee directories
  • • Meeting minutes

Protection: Access controls, VPN

3

Confidential

Sensitive business information requiring protection. Restricted access.

Examples:

  • • Financial statements
  • • Customer databases
  • • Strategic plans
  • • Contracts & agreements

Protection: Encryption, MFA, audit logs

4

Restricted/Secret

Highest sensitivity. Severe impact if disclosed. Strict controls.

Examples:

  • • PII (Personally Identifiable Info)
  • • Trade secrets
  • • Intellectual property
  • • M&A plans

Protection: End-to-end encryption, DLP, need-to-know
